# ERP-MES Printing (ISA-95)

Printing industry (flexo) **overlay** on top of the industry-neutral
[`erp-mes-core`](../erp-mes-core/README.md) bundle. It turns the canonical ISA-95 backbone into a
concrete MES for a printing house: flexo press fleet, film/ink materials, the Flexibase customer
event code catalog (155 codes, OGP/BM/POUCH/MES), Zero Pull setup tags, printing quality
dictionaries and a seeded demo print order with job bag (work record) sections.

The bundle is generated by `generate_bundle.py` (single source of truth). Never edit `bundle.json`
by hand — change the generator and re-run:

```bash
python examples/erp-mes-printing/generate_bundle.py
```

`event_catalog_data.py` next to the generator is reference data parsed one-off from the Flexibase
customer SQL seeds (`downtime_event_code_dict_customer.sql`, `downtime_setup_tag_dict_zp.sql`) —
do not edit by hand either.

## Dependency on erp-mes-core

- Top-level `requires: [{"appId": "erp-mes-core", "minVersion": "1.1.0"}]` — deploy **the core
  first**, the platform dependency verifier rejects the overlay otherwise.
- Same `schemaName` as the core (`app_erp_mes_core`): overlay migrations, functions and bindings
  run with `search_path` into the core schema and read/write the core `emc_*` tables directly.
- Own `tablePrefix: emp_` — only the two overlay tables are created by this bundle; everything
  else is data seeded into the core tables (idempotent `INSERT ... SELECT ... WHERE NOT EXISTS`).
- Migration ids use the `emp_m` prefix (global `root.platform.migrations` namespace).
- Overlay BFF functions use the `emp_` prefix — core `emc_*` function names are never reused
  (function lookup is global and would shadow the core implementations).

## Deploy

Two steps — the core first, then the overlay:

```bash
# 1. login (default dev credentials)
TOKEN=$(curl -s -X POST http://localhost:8080/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"admin"}' | python -c "import sys,json;print(json.load(sys.stdin)['token'])")

# 2. deploy the core (required)
curl -s -X POST http://localhost:8080/api/v1/applications/erp-mes-core/deploy \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  --data-binary @examples/erp-mes-core/bundle.json

# 3. deploy the overlay
curl -s -X POST http://localhost:8080/api/v1/applications/erp-mes-printing/deploy \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  --data-binary @examples/erp-mes-printing/bundle.json

# 4. call an overlay BFF function
curl -s -X POST http://localhost:8080/api/v1/bff/invoke \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  -d '{"objectPath":"root.platform.singleton-blueprints.erp-mes-printing-hub-v1","functionName":"emp_eventdef_list",
       "input":{"schema":{"name":"in","fields":[{"name":"section","type":"STRING"},{"name":"catalog","type":"STRING"}]},
                "rows":[{"section":"","catalog":"OGP"}]}}'
```

Then open **Solutions → ERP-MES Printing (ISA-95)** in the web console.

## Happy path (seeded demo data)

1. Open the **Диспетчер печати** dashboard — `JO-PRINT-001` is RUNNING on PR120 (flexo print),
   `JO-PRINT-002` is ALLOWED on LM210, `JO-PRINT-003` is NOT_ALLOWED on SL300.
2. Register a setup event with the **Зарегистрировать событие** form on **События и простои**
   (code `OGP-120` picked from the `emp-eventdef-catalog` dropdown, job `JO-PRINT-001`) —
   the core `emc_event_register` accepts overlay catalog codes; AVAILABILITY events stay OPEN and
   raise the `openPrintDowntimeCount` hub counter.
3. Stage materials on the **Рулоны и материалы** dashboard: place roll `BC-FILM-0001` (PET 12 µm,
   1050 mm) on `JO-PRINT-001`, consume metres — the core writes Material Actuals and genealogy.
4. Start `JO-PRINT-002` (lamination) once the printed roll is ready — the resource-conflict guard
   of the core allows it (different work unit).
5. Review downtime in **События и простои** (`emp-downtime-by-code` aggregates by Flexibase code
   and section); filter the catalog with `emp_eventdef_list` (section `Printing`, catalog `OGP`).
6. The job bag: `emc_wrec_get` for `JO-PRINT-001` returns `colorControl` (ΔE2000, viscosity, Lab)
   and `imposition` (cut-off, registration, plate set) sections; add your own via
   `emc_wrec_saveSection`.

## Conscious gaps (by design)

- No own lifecycle functions — job orders, materials, QA, events, calendar and OEE are driven by
  the **core** `emc_*` functions on purpose (the overlay is data + analytics + UI only).
- Zero Pull tags are a plain dictionary (`emp_setup_tag`); linking them to individual setup events
  (tag per repeated attempt) is a project-level extension.
- No ERP connector for the printing order flow — the core Part 5 outbox/inbox is reused as-is.
- Devices mirror only three work units (PR120/LM210/SL300); PR130 and RW100 live in the ISA-95
  equipment model without a live object.
- Multi-segment routing (print → laminate → slit as one Work Definition) is not modelled — the
  core Work Master is single-segment; chained job orders carry the flow.

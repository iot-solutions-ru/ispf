# ERP-MES Core (ISA-95)

Canonical [ISA-95 / IEC 62264](https://www.isa.org/standards-and-publications/isa-standards/isa-standards-committees/isa95)
foundation bundle for ISPF. It gives you the Level 3 ↔ Level 4 backbone — equipment/material/personnel
models, work definitions, scheduling, dispatching, execution, genealogy, quality, inventory documents,
maintenance-lite, shift calendar, OEE and ERP integration transactions — so that concrete ERP/MES
products can be assembled on top in days, not months.

The bundle is generated by `generate_bundle.py` (single source of truth). Never edit `bundle.json` by hand —
change the generator and re-run:

```bash
python examples/erp-mes-core/generate_bundle.py
```

## What is inside

| ISA-95 part | Concept | Tables (prefix `emc_`) | BFF functions |
|---|---|---|---|
| Part 1/2 | Equipment hierarchy (Enterprise→Site→Area→Work Center→Work Unit, Storage Zone/Unit), equipment classes & properties | `emc_equipment`, `emc_equipment_class`, `emc_equipment_property` | `emc_equipment_list` |
| Part 2 | Personnel model, classes, qualifications | `emc_person`, `emc_personnel_class`, `emc_person_qualification` | `emc_person_list` |
| Part 2 | Material model: classes, definitions, lots/sublots, lot properties | `emc_material_class`, `emc_material_definition`, `emc_material_lot`, `emc_material_sublot`, `emc_material_lot_property` | `emc_matlot_register/get/list/placeOnLine/removeFromLine/consume/produce`, `emc_stock_list` |
| Part 2 | Process Segment + segment specifications (material/equipment/personnel) — the canonical "operation BoM/routing" | `emc_process_segment`, `emc_segment_material_spec`, `emc_segment_equipment_spec`, `emc_segment_personnel_spec` | `emc_segment_list` |
| Part 4 | Work Definition (Work Master, versioned) | `emc_work_master` | `emc_workmaster_upsert/list` |
| Part 4 | Work Schedule → Work Request → Job Order (+ requirement snapshots, audit) | `emc_work_schedule`, `emc_work_request`, `emc_job_order`, `emc_job_order_*_req`, `emc_job_order_audit` | `emc_schedule_receive/list`, `emc_joborder_listBoard/release` |
| Part 4 | Work Performance → Job Response + actuals (material/equipment/personnel), response data (RUN/PAUSE intervals, PDC) | `emc_job_response`, `emc_job_response_data`, `emc_material_actual`, `emc_equipment_actual`, `emc_personnel_actual` | `emc_joborder_start/pause/resume/complete/abort/replan/confirmStart`, `emc_dc_recordQuantity` |
| Part 4 | Production Tracking & Genealogy | `emc_lot_genealogy` | `emc_track_genealogyByLot`, `emc_track_jobHistory` |
| Part 3 | Quality Operations: defect types, reason codes, defect workflow (REGISTERED→CONFIRMED/REJECTED→CLOSED), test results | `emc_defect_type`, `emc_reason_code`, `emc_defect_record`, `emc_defect_status_history`, `emc_qa_test_result` | `emc_qa_registerDefect/confirmDefect/rejectDefect/closeDefect/listDefects/recordTestResult` |
| Part 3 | Inventory Operations: canonical movement documents | `emc_inventory_document`, `emc_inventory_document_line` | `emc_invdoc_create/addLine/submit/apply/list` |
| Part 3 | Maintenance Operations (lite): request → work order | `emc_maintenance_request`, `emc_maintenance_work_order` | `emc_maint_createRequest/acceptRequest/completeWorkOrder/list` |
| Part 2 (2018) | Operations Event Definition/Event (downtime & OEE loss capture, six big losses), raw L2 signals | `emc_operations_event_definition`, `emc_operations_event`, `emc_machine_signal` | `emc_eventdef_list`, `emc_event_register/registerSignal/close/list` |
| Part 4 | Work Calendar (shifts) & assignments | `emc_work_calendar`, `emc_shift_assignment` | `emc_calendar_openShift/closeShift/assignPerson/current` |
| Part 4 cl.15 | Work Record (production dossier / job bag) | `emc_work_record`, `emc_work_record_section` | `emc_wrec_get/saveSection` |
| Part 5 | B2M transactions: verb × noun outbox/inbox, idempotency keys, normalized ACK, integration log, master-data replicas | `emc_erp_outbox`, `emc_erp_inbox`, `emc_integration_log`, `emc_master_data_replica` | `emc_erp_enqueueOutbox/pollOutbox/receiveInbox/syncMasterData/listOutbox` |
| ISO 22400 | OEE per work unit per shift (A×P×Q) | `emc_oee_shift` | `emc_oee_calcShift/getKpi/listShifts` |

Job-order state machine follows OPC 10031-4 (ISA-95 job order states):
`NOT_ALLOWED → ALLOWED → RUNNING ⇄ SUSPENDED(PAUSED) → ENDED | ABORTED | CANCELLED`.

Warehouses are modeled canonically as **equipment** of level `STORAGE_ZONE` / `STORAGE_UNIT`
(seeds: `WH-CENTRAL`, `WH-LINE-A01`), lots reference them via `storage_location`.

## Live objects & UI

- Hub singleton node `root.platform.singleton-blueprints.erp-mes-core-hub-v1` (SINGLETON blueprint
  `erp-mes-core-hub-v1`, live node auto-created by bundle deploy): KPI counters
  (`pendingOutboxCount`, `activeDowntimeCount`, `lowStockCount`) fed by SQL bindings, 2 alert rules,
  2 (disabled) schedules, event journal, and all 61 BFF functions.
- Two work-unit devices `emc-wu-a01` / `emc-wu-a02` (blueprint `emc-work-unit-v1`): `status`, `speed`,
  `activeJobOrderId` mirrored by job-order lifecycle.
- 5 dashboards (operator UI in Russian): **Диспетчер** (job board + lifecycle + KPI cards),
  **Исполнение** (consume/place/produce/PDC), **Склад** (stock + ERP docs), **Качество** (defect
  workflow), **OEE и простои** (event journal + shift KPIs). Widgets use the flat dashboard format
  (`id`/`type`/settings inline); form fields are dropdown selects fed by catalog reports
  (`optionsFromReport`), report rows write `session.params` on click (`rowParamsFromRowJson` +
  `autoSelectFirstRow`) so the forms pick up the selected job/lot/defect automatically.
- 13 reports: 5 operational (`emc-job-board`, `emc-stock-report`, `emc-material-movement`,
  `emc-defect-report`, `emc-oee-shift-report`) + 8 catalog/option sources
  (`emc-material-catalog`, `emc-equipment-catalog`, `emc-person-catalog`, `emc-defect-type-catalog`,
  `emc-reason-code-catalog`, `emc-eventdef-catalog`, `emc-shift-catalog`, `emc-event-journal`)
  reused by overlay bundles for dropdowns.
- 1 BPMN workflow (`emc-job-dispatch` — operator confirms job start), 2 platform events.

## Deploy

```bash
# 1. login (default dev credentials)
TOKEN=$(curl -s -X POST http://localhost:8080/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"admin"}' | python -c "import sys,json;print(json.load(sys.stdin)['token'])")

# 2. deploy the bundle
curl -s -X POST http://localhost:8080/api/v1/applications/erp-mes-core/deploy \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  --data-binary @examples/erp-mes-core/bundle.json

# 3. call a BFF function (wireProfile ispf-operator-v1)
curl -s -X POST http://localhost:8080/api/v1/bff/invoke \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  -d '{"objectPath":"root.platform.singleton-blueprints.erp-mes-core-hub-v1","functionName":"emc_joborder_listBoard",
       "input":{"schema":{"name":"in","fields":[]},"rows":[{}]}}'
```

Then open **Solutions → ERP-MES Core (ISA-95)** in the web console.

## Happy path (seeded demo data)

1. `emc_schedule_receive` — ERP sends an order (creates Work Schedule → Request → Job Order `NOT_ALLOWED`
   with requirement snapshots from segment specs).
2. `emc_joborder_release` → `ALLOWED`; `emc_joborder_start` → `RUNNING`
   (guards: resource conflict per work unit; writes Job Response + RUN interval + actuals;
   emits Part 5 `PROCESS OperationsEvent` "work commenced" to the outbox).
3. `emc_matlot_placeOnLine` (barcode) → lot `ON_LINE`; `emc_matlot_consume` / `emc_matlot_produce`
   write Part 4 Material Actuals and genealogy edges automatically.
4. `emc_qa_registerDefect` → confirm → `emc_joborder_complete` is **blocked** (`QC_GATE_BLOCKED`)
   until the defect is closed — the canonical quality gate.
5. `emc_joborder_complete` → `ENDED` + Part 5 `PROCESS OperationsPerformance` "work completed" outbox message.
6. `emc_erp_pollOutbox` (simulator) → normalized `ACK ACCEPTED`; `emc_erp_receiveInbox` handles inbound
   `OPERATIONS_SCHEDULE` / `MASTER_DATA` nouns idempotently.
7. `emc_oee_calcShift` → A/P/Q/OEE per shift (events bucketed by `oee_bucket`, produced qty from actuals,
   defects from QA records).

Seeded demo landscape: hierarchy `ENT-DEMO/SITE-01/AREA-PROD/LINE-A` with work units `WU-A01` (assembly),
`WU-A02` (packing); segments `SEG-ASSEMBLE`, `SEG-PACK` with full specs; work masters `WM-ASSEMBLE`/`WM-PACK`;
materials RAW→WIP→FG with lots and barcodes; schedule `SCH-DEMO-001` with `JO-DEMO-001` (ALLOWED),
`JO-DEMO-002` (RUNNING on WU-A01), `JO-DEMO-003` (ALLOWED, resource-conflict demo); shift `SHIFT-DEMO-1`.

## Extending for a real project (the overlay pattern)

The core is deliberately industry-neutral. A concrete MES/ERP product = this bundle + a small overlay:

1. **Master data** — insert your equipment hierarchy, materials, segments and work masters
   (seeds or `emc_workmaster_upsert`). No new tables needed: extension points are
   `emc_equipment_property`, `emc_material_lot_property`, segment specifications, event definitions
   and work-record sections — all data-driven.
2. **Event definitions** — add your downtime/reason codes as rows in `emc_operations_event_definition`
   (map to `oee_bucket` + six big losses); OEE picks them up automatically.
3. **Work record sections** — `emc_wrec_saveSection` builds the production dossier (job bag) per job order;
   sections are free-form JSON keyed by `section_key`.
4. **Industry overlay** — e.g. printing (docanima-style): segments `PRINT`/`CUT`/`BIND`, materials
   `PAPER`/`INK`/`PLATE`, event codes `MAKEREADY`/`WASH_UP`, work-record sections `colorControl`, `imposition`.
   Same tables, same functions, zero schema changes.
5. **ERP connector** — replace the outbox simulator with a real transport by polling `emc_erp_outbox`
   (`status='PENDING'`) and writing ACKs; inbound ERP calls go through `emc_erp_receiveInbox`.

## Conscious gaps (by design)

- B2MML XML payloads — the integration layer carries canonical verb×noun semantics in JSON;
  XML (B2MML V0600/V0701 XSD) mapping is a connector concern.
- Physical Asset model (Part 2 2018) — equipment model covers it for MES scope.
- Operations Capability / Process Segment Capability queries (Part 4 cl.10) — not yet surfaced as functions.
- Workflow Specification graph (Part 4 cl.7) — Work Master is single-segment; multi-segment routings
  are an overlay.
- Full ISO 22400 KPI catalogue — only the OEE subset (A/P/Q/OEE) is computed.
- APS-grade finite scheduling, full CMMS — out of scope (Maintenance is lite by design).

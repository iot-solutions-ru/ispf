# ERP-MES Pharma (ISA-95/ISA-88)

Pharmaceutical (solid dosage) **overlay** on top of the industry-neutral
[`erp-mes-core`](../erp-mes-core/README.md) bundle. It turns the canonical ISA-95 backbone into a
GMP-flavoured MES for a solid-dosage plant: granulation → drying → blending → tableting → coating →
blistering → cartoning line model, API/excipient/packaging materials with batch genealogy,
quarantine → release lot disposition with electronic signatures (21 CFR Part 11 style), double
verification of dispensing weighments, an electronic batch record (eBR) skeleton and DataMatrix
serialization with aggregation.

The bundle is generated by `generate_bundle.py` (single source of truth). Never edit `bundle.json`
by hand — change the generator and re-run:

```bash
python examples/erp-mes-pharma/generate_bundle.py
```

## Dependency on erp-mes-core

- Top-level `requires: [{"appId": "erp-mes-core", "minVersion": "1.0.0"}]` — deploy **the core
  first**, the platform dependency verifier rejects the overlay otherwise.
- Same `schemaName` as the core (`app_erp_mes_core`): overlay migrations, functions and bindings
  run with `search_path` into the core schema and read/write the core `emc_*` tables directly.
- Own `tablePrefix: pha_` — only the three overlay tables are created by this bundle; everything
  else is data seeded into the core tables (idempotent `INSERT ... SELECT ... WHERE NOT EXISTS`).
- Migration ids use the `pha_m` prefix (global `root.platform.migrations` namespace).
- Overlay BFF functions use the `pha_` prefix — core `emc_*` function names are never reused
  (function lookup is global and would shadow the core implementations).

## GMP specifics modelled here

- **Quarantine → release workflow.** Every produced/purchased lot lands in `QUARANTINE`
  (`WH-QUAR`). `pha_lot_release` moves it to `RELEASED` (status `STOCK`) and writes an `APPROVED`
  e-signature; `pha_lot_reject` moves it to `REJECTED` (`BLOCKED_QC`) with a `REVIEWED` signature.
  Both guard the source disposition — a second release attempt returns `INVALID_STATE`.
- **Electronic signatures (21 CFR Part 11 style).** `pha_esign_record` stores
  entity/signer/meaning/comment/timestamp; `pha_esign_list` and the `pha-esign-audit` report give
  the audit trail. Meanings: `AUTHORED`, `REVIEWED`, `APPROVED`.
- **Double verification of weighments.** `pha_dispense_weigh` enforces two different persons
  (`SECOND_PERSON_REQUIRED` otherwise), checks the deviation against a tolerance
  (`TOLERANCE_EXCEEDED`) and records both signatures (`AUTHORED` + `REVIEWED`).
- **eBR skeleton.** The work record of `JO-PH-001` carries GMP sections (dispensing protocol, IPC
  results, line clearance, deviations); extend with the core `emc_wrec_saveSection`.
- **Serialization.** `pha_serial_register` commissions DataMatrix codes with GTIN/lot/job
  references and a `parentCode` for aggregation (unit → box → pallet); duplicates are rejected with
  `DUPLICATE_SERIAL`.

## Deploy

Two steps — the core first, then the overlay:

```bash
# 1. login (default dev credentials)
TOKEN=$(curl -s -X POST http://localhost:8080/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"admin"}' | python -c "import sys,json;print(json.load(sys.stdin)['token'])")

# 2. deploy the core (required)
curl -s -X POST http://localhost:8080/api/v1/applications/erp-mes-core/deploy \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  --data-binary @examples/erp-mes-core/bundle.json

# 3. deploy the overlay
curl -s -X POST http://localhost:8080/api/v1/applications/erp-mes-pharma/deploy \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  --data-binary @examples/erp-mes-pharma/bundle.json

# 4. call an overlay BFF function
curl -s -X POST http://localhost:8080/api/v1/bff/invoke \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  -d '{"objectPath":"root.platform.singleton-blueprints.erp-mes-pharma-hub-v1","functionName":"pha_eventdef_list",
       "input":{"schema":{"name":"in","fields":[{"name":"section","type":"STRING"}]},
                "rows":[{"section":""}]}}'
```

Then open **Solutions → ERP-MES Pharma (ISA-95/ISA-88)** in the web console.

## Happy path (seeded demo data)

1. Open the **Производство (фарма, ISA-88)** dashboard — `JO-PH-001` is RUNNING on TPR-01
   (tableting), `JO-PH-002` is ALLOWED on COT-01, `JO-PH-003` is NOT_ALLOWED on BLS-01.
2. Dispense on the **Взвешивание** form: scan the barcode of an excipient lot (`BC-EXC-0001`),
   enter target/actual weight and two different operators — both e-signatures are recorded.
3. On **Качество и выпуск серии (GMP)**: two lots sit in quarantine (`quarantineLotCount = 2`
   raises the `quarantineAgingAlert`). Release `LOT-FG-PH-0001` with a QA signature — the lot becomes
   `RELEASED/STOCK` and the audit report gains an `APPROVED` row; a repeated release is refused.
4. Register a deviation (defect) with the core QA form — `openDeviationCount` goes up and raises
   `deviationOpenAlert`.
5. On **Сериализация и аудит**: register a new DataMatrix code against `JO-PH-003` with a box
   aggregate in `parentCode`; re-submitting the same code returns `DUPLICATE_SERIAL`.
6. The eBR: `emc_wrec_get` for `JO-PH-001` returns the `lineClearance` and `ipcResults` sections.

## Conscious gaps (by design)

- No real interface to the national serialization registry («Честный знак» / EPCIS) or to a LIMS —
  `pha_serial_record` and the e-sign journal are the system of record in this demo.
- IPC/CIP/SIP activities are represented as event codes and eBR sections, not as automated
  equipment phases — no batch engine (ISA-88 procedural control) is attached.
- No validation documentation package (IQ/OQ/PQ), no audit-trail review workflow — out of scope
  for a reference bundle.
- Job lifecycle stays in the **core** `emc_*` functions; the overlay adds only GMP-specific
  transaction guards (disposition, double verification, serialization).

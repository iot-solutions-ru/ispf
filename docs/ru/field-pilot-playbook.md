# Playbook полевого пилота (БЛ-140)

Повторяемые инструкции для полевых пилотов OT для трех эталонных сценариев. Каждый пилотный проект проверяет **ПРОИЗВОДСТВЕННЫЕ** драйверы, записывает данные туда и обратно, принимает архивные данные и использует HMI оператора без промежуточного программного обеспечения между ISPF и периферией предприятия.

Предварительные требования: ISPF ≥ 0.9.32, [DRIVER_INTEROP_LAB.md](driver-interop-lab.md) зелёный локально, учетная запись оператора с ролью администратора устройства.

---

## Пилотная матрица

| Сценарий | Основной драйвер | Типичный сайт | Критерии успеха | Статус |
| -------- | -------------- | ------------ | ---------------- | ------ |
| **Завод Modbus** | `modbus-tcp` | Резервуарный парк PLC/RTU, насосные станции | опрос более 50 тегов <2 с; FC16 записывает туда и обратно; live-симуляция | **Готов к полевым работам** |
| **Парк MQTT** | `mqtt` | Распределенные шлюзы, пакетная телеметрия | 10+ устройств на общем брокере; подписка на вход; сигнализация о несвежем | **Готов к полевым работам** |
| **Линия OPC UA** | `opcua` + `opcua-server` | Ячейки упаковочной/сборочной линии | Просмотр + подписка; внешняя обратная запись UA; Имитатор SCADA | **Готов к полевым работам** |

Цель выдержки: **минимум 7 дней** на пилота; регистрировать инциденты в пилотном журнале (см. § Закрытие).

## 1 — Пилотная установка Modbus

### Объем

Одиночный узел ISPF в VLAN предприятия, считывающий ПЛК Modbus TCP (или приспособление `deploy/driver-interop` для пробного запуска).

### Настройка устройства

| Настройка | Пример |
| ------- | ------- |
| `driverId` | `modbus-tcp` |
| `host` | `192.168.10.50` |
| `port` | `502` |
| `pollIntervalMs` | `1000` |
| `unitId` | `1` |

Сопоставьте регистры хранения переменным (`40001` → `tank.level` и т. д.). См. [DRIVERS.md](drivers.md) § modbus-tcp.

### Контрольный список проверки

1. **Подключиться** — состояние устройства `RUNNING`; нет `DriverException` в журналах.
2. **Чтение** — обновление 50 тегов за два цикла опроса; качество `GOOD`.
3. **Write** — оператор или API `PUT` на записываемую катушку/регистр; ПЛК подтверждает изменение (или прибор обратной связи выдает эхо).
4. **Историк** — `GET /api/v1/variables/{path}/history` возвращает образцы для 5 самых популярных тегов.
5. **HMI** — мнемосхема или дашборд отображают текущие значения; устаревший значок после отключения брокера/ПЛК.

### Пробный прогон (лаборатория)

```bash
docker compose -f deploy/driver-interop/docker-compose.yml up -d --wait
bash deploy/tools/driver-interop-smoke.sh
./gradlew :packages:ispf-driver-modbus:test
```

### Откат

Остановить выполнение драйвера, вернуть снимок конфигурации устройства, восстановить программу ПЛК, если тесты записи коснулись заданных значений.

## 2 — Пилот флота MQTT

### Объем

Группа пограничных шлюзов, публикующих телеметрию JSON для общего брокера (Mosquitto, HiveMQ или облачного MQTT).

### Настройка устройства

| Настройка | Пример |
| ------- | ------- |
| `driverId` | `mqtt` |
| `brokerUrl` | `tcp://mqtt.plant.local:1883` |
| `clientId` | `ispf-fleet-01` |
| `topicPrefix` | `plant/line1/{deviceId}/telemetry` |

Одно устройство ISPF на шлюз или мультиплексированная гибкая тематическая карта — см. [DRIVERS.md](drivers.md) § mqtt.

### Контрольный список проверки

1. **Вход** — 10 устройств онлайн; Количество переменных соответствует карте тем.
2. **Пакет** — симулирует скорость 1 000 сообщений/мин в течение 10 минут; входная очередь не теряется (диагностика `/api/v1/drivers/runtime`).
3. **Запись** — команда публикации темы (заданное значение/подтверждение) возвращается в рамках SLA.
4. **Тревога** — устаревшее правило срабатывает, когда устройство прекращает публикацию > 3× `pollIntervalMs`.
5. **Историк** — экспорт тенденций в формате CSV для одного скоростного тега.

### Пробный прогон (лаборатория)

```bash
docker compose -f deploy/driver-interop/docker-compose.yml up -d --wait
mosquitto_pub -h 127.0.0.1 -t ispf/lab/ping -m ok
./gradlew :packages:ispf-driver-mqtt:test
```

### Откат

Отключите правила сигналов тревоги, остановите устройства MQTT, очистите темы тестирования на брокере.

## 3 — пилот линии OPC UA

### Объем

Конвейер с серверами OPC UA на ячейках; ISPF в качестве агрегатора и опционального экспортера `opcua-server` для партнеров MES/SCADA.

### Настройка устройства (клиент)

| Настройка | Пример |
| ------- | ------- |
| `driverId` | `opcua` |
| `endpointUrl` | `opc.tcp://192.168.20.10:4840` |
| `readMode` | `subscribe` |
| `timeoutMs` | `10000` |

Сопоставление точек: `ns=2;s=LineSpeed` и т. д. SecurityPolicy Нет для лаборатории; подписать + зашифровать для рабочей VLAN.

### Настройка устройства (экспорт с сервера)

| Настройка | Пример |
| ------- | ------- |
| `driverId` | `opcua-server` |
| `bindPort` | `4840` |
| `namespace` | `2` |

Предоставьте выбранные переменные ISPF внешним клиентам UA (UA Expert, Prosys). См. [OPCUA_SERVER_INTEROP.md](opcua-server-interop.md).

### Контрольный список проверки

1. **Обзор** — отображается `IspfVariables` папка или пространство имен ячеек.
2. **Подписаться** — отправлять обновления без отката опроса; задержка < 500 мс в локальной сети лаборатории.
3. **Обратная запись** — запись внешнего UA-клиента обновляет переменную ISPF (шаблон `OpcUaServerSubscriptionWriteBackIntegrationTest`).
4. **Чтение партнера** — сторонний клиент подписывается на конечную точку `opcua-server`.
5. **Mimic** — мнемосхема обзора строки, привязанная к подписанным тегам.

### Пробный прогон (лаборатория)

```bash
./gradlew :packages:ispf-driver-opcua-server:test
./gradlew :packages:ispf-driver-opcua:test
```

### Откат

Stop `opcua-server` device (frees port 4840), revert client endpoint URLs.

**Определение готовности (BL-140):** три пилотных проекта **готовы к эксплуатации**, имеют зелёный цвет лабораторных пробных испытаний, заполнены контрольные списки и опубликован шаблон подписания; 7-дневная задержка клиента — это оставшиеся gates до подтверждения продвижения. Связность ОТ по конкурентной системе показателей ≥ 9,5 (после волны 8).
